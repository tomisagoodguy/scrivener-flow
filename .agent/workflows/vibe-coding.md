# Vibe Coding 技巧指南

本指南整合了台大 NTU Web Programming 課程與吳恩達 (Andrew Ng) 的 Vibe Coding 核心精神，旨在教導如何轉變角色，從「打字員」升級為「AI 主管」。

## 核心心法：從 Coder 轉向 Supervisor

1. **重新定義程式能力**：
   - 現代程式能力 = **需求分析** + **系統邏輯** + **任務轉換（Task Breakdown）**。
   - 不再糾結於語法細節，而是專注於如何「指揮」一群不會累的「Super Junior」AI 工程師。

2. **Top-Down 學習法**：
   - **先產出再理解**：讓 AI 先產出 Code (One-shot)，然後再閱讀與微調。這比查文件（Try-and-error）快 10 倍。
   - **看懂而非手寫**：不需要背誦 API 參數，但要能看懂 AI 產出的邏輯、結構與潛在 Bug。

3. **代理人模式 (Agentic Workflow)**：
   - 利用背景執行（Background agents）同時跑多個想法。
   - 採用「收割模式」：晚上睡前丟出多個 Idea 交給 AI 代理人跑，隔天早上收割完成 70-80% 的成果。

## 提示詞五大要素 (Andrew Ng)

當你需要 AI 建立功能或應用程式時，盡量在提示詞中包含以下要素：

| 要素 | 說明 | 範例 |
| :--- | :--- | :--- |
| **目標 (Goal)** | 你最終想創造什麼 | 「我想做一個能幫我寫生日卡片的 Web App」 |
| **輸入 (Input)** | 使用者需要提供什麼資料 | 「輸入姓名、年齡、生肖、愛好」 |
| **佈局 (Layout)** | 視覺呈現與元素位置 | 「輸入框在左邊，產出的卡片預覽在右邊，要有節慶感的深色主題」 |
| **特殊功能 (Special Features)** | 額外的進階行為 | 「點擊可以切換多種語氣（幽默、感人、正式）」 |
| **輸出 (Output)** | 最終產出的結果是什麼 | 「回傳一段有趣的訊息並生成可下載的圖片」 |

## 開發策略

### 1. 單一長提示 (Single Long Prompt) vs. 增量提示 (Incremental Prompting)

- **單一長提示**：當規格清晰時使用，能產出更高質量的「版本 1.0」。
- **增量提示**：對概念還模糊、或功能極度複雜時，先建骨架，再一塊塊「長」出肌肉。

### 2. 回饋與迭代 (Feedback Loop)

- 不要期待 AI 一次就答對。把 AI 產出的錯誤當作學習與校準的過程。
- 分步溝通：先確認邏輯流程 (Logic Flow)，再請 AI 實作細節。

### 3. 工具選擇 (Tools)

- 強制使用專門為 AI 設計的 IDE (如 Cursor, Windsurf)，而非單純的網頁聊天介面。
- 善用程式庫的 Context：透過 `@file`, `@docs`, `@folder` 提供精確上下文，減少 AI 幻覺。

## 實作進階技巧

1. **增量交付 (Incremental Delivery)**：
   - 不要一次要求 AI 完成整個大專案。
   - 先求「能跑」(PoC)，再求「美化」(UI/UX)，最後求「優化」(Performance/Security)。

2. **上下文定錨 (Context Anchoring)**：
   - 在開始新任務前，先叫 AI 閱讀專案的 `README.md`, `CLAUDE.md` 或現有的代碼結構。
   - 確保 AI 的「Vibe」與專案現有的風格一致。

3. **快速重啟 (Fail Fast & Restart)**：
   - 如果 AI 開始在同一個 Bug 上打轉超過 3 次，不要繼續在那條對話紀錄裡糾纏。
   - 開一個新對話 (New Chat)，重新梳理需求，並把剛才失敗的經驗整理成「不要這樣做」的指令。

4. **收割與驗證 (Harvesting & Verification)**：
   - AI 寫完後，第一時間不是看 Code，而是跑跑看。
   - 確認「功能感 (Vibe)」對了，再回頭檢視代碼質量。

## 學習建議

- 既然電腦語言是電腦在講的，我們只要知道它在講什麼就好。
- 人類記憶有限，專注於那些「不會隨版本更新而過時」的系統架構與產品思維。
- 把 AI 產出的重複性工作自動化，把自己的精神留在最關鍵的核心決策。
- 養成記錄 `.agent/LEARNINGS.md` 的習慣，這就是你與 AI 共同累積的「經驗值」。

## 教練介入規則 (Coaching Triggers)

當偵測到以下情況時，AI 必須主動介入提示：

1. **細節糾結 (The Syntax Trap)**：
    - *觸發*：使用者試圖手寫複雜的 Regex、CSS 漸層、或是瑣碎的 API 調用細節。
    - *提示*：提示使用者「只需定義 Vibe/邏輯」，其餘細節交給 AI 自動完成。

2. **低效迭代 (The Loop Trap)**：
    - *觸發*：在同一個錯誤或 Bug 上連續對話超過 3 次且無進展。
    - *提示*：建議結束當前對話，清空過時上下文，並開新對話重申 Goal。

3. **上下文缺失 (The Context Void)**：
    - *觸發*：使用者給出模糊需求（如「幫我改一下樣式」）而未指定文件、代碼或目標風格。
    - *提示*：主動要求使用 `@file` 或指定現有文件作為參考點藍圖。

4. **捨近求遠 (The Bottom-Up Struggle)**：
    - *觸發*：試圖從零開始建構大型功能，而不是先要一個 PoC 並以此進行迭代。
    - *提示*：建議改用「Top-Down」模式，先拿產出再微調。
